# drama-forge 开发记录与方法论

这个文件回答一个问题:一个人如何借助 AI 编程工具,在 5 天内从零做出一个可用、可演示、可开源的完整工具。

## 总体流程

```
行业调研 → 设计文档(唯一权威) → 分阶段开工提示词 → AI 建造(逐任务循环) → 实跑验收 → 下一阶段
```

核心原则只有三条:

1. **文档给全,开工分阶段。**设计文档一次写完整(视觉 token、每页规格、数据流、安全规则、演示预案),是所有阶段的唯一依据;开工提示词一次只发一个阶段,上一阶段验收通过才写下一份,避免在错误的地基上盖楼。
2. **执行循环标准化。**每个阶段的提示词都包含同一套纪律:逐任务「实现 → 自检(node --check + 启动冒烟)→ commit」;不重构已验收代码;卡住列方案等确认,不硬试;完成只交文字报告。
3. **验收看真实运行。**每阶段验收不看代码看效果:启动应用,把该阶段的核心路径实际点一遍。最终验收是一条「黄金路径」:断网 + 演示模式,从首页热点 → AI 选题 → 带参进文案工厂流式生成 → 分镜 → 周报 → 提示词库 → 历史记录,全程无报错、无生硬感。

## 提示词工程的沉淀

`prompts/` 目录下每个模板的头部注释都写了设计思路,这里补充几条通用经验:

- **输出格式要为解析而设计。**文案生成用 `===TITLES===` 这类固定分隔标记,前端才能做流式分组渲染;分镜强制「只输出 JSON 数组」,后端配合容错解析(剥围栏、截取方括号、失败重试一次)。让 AI 的自由发挥收敛在结构里。
- **平台差异写进提示词而不是 UI 文案。**抖音=短句悬念前置,快手=口语化强情绪,微信小程序=剧情完整+付费钩子。差异要真实到「换个平台重新生成,外行都能看出味道不一样」。
- **约束比措辞重要。**周报模板里「数字必须原样保留、没提到的不编造」这两句,比任何风格描述都更能保证输出可信。

## 工程上的踩坑与对策

- **流式超时不能用总闹钟。**长生成会被固定超时掐断,改成「空闲超时」:有数据就续命,30 秒没新数据才算失败。
- **免费热榜接口不可信任。**必须逐个实测,每个适配器留主备两个候选;测不通的源在界面上如实显示「暂不可用」,不硬编码假数据糊弄。缓存和演示模式是最后两层兜底。
- **动用户文件的功能,安全设计优先于功能设计。**先预览后执行、只在选定目录内移动、路径穿越校验、重名自动加后缀、完整日志加一键撤销、绝不删除任何文件。
- **CSV 导出必须带 UTF-8 BOM**,否则 Excel 打开中文乱码。
- **演示的容错是产品的一部分。**热点缓存、演示模式、假流式,让工具在断网的会议室里也能完整演示——这本身就是「确保工具稳定运行」的答卷。

## 工具分工

- **Claude(Claude Code)**:行业调研、设计文档、全部六份阶段提示词、阶段 5~6 直接编码、每阶段验收诊断。
- **Codex**:按阶段提示词执行阶段 1~4 的建造,逐任务 commit。
- **自动化测试**:node:test,零依赖,共 91 个用例,覆盖配置、热榜缓存、AI 层、文件安全、提示词库、历史、mock 数据质量门槛(条数/结构/时长约束直接写成断言)。
